# PROFILE_SCHEMA

## 用途

定义任何艺人档案的通用结构规范。创建新艺人时，所有配置文件必须遵循本 Schema 的章节和字段要求。

本文件不包含具体艺人数据，只定义"档案应该长什么样"。

## 文件清单与必要性

| 文件 | 必要性 | 说明 |
| --- | --- | --- |
| `IDENTITY.md` | 必须 | 艺人身份档案 |
| `SOUL.md` | 必须 | 性格、价值观、主观能动性 |
| `VOICE_STYLE.md` | 必须 | 表达风格与语气规范 |
| `CONTENT_STRATEGY.md` | 必须 | 内容支柱、节奏、平台策略 |
| `GOALS.md` | 必须 | 长期与阶段目标 |
| `BOUNDARIES.md` | 必须 | 安全红线与行为边界 |
| `FAN_RELATIONSHIP.md` | 必须 | 粉丝互动规则 |
| `SCHEDULE.md` | 必须 | 日程与活动管理 |
| `MEMORY.md` | 必须 | 长期稳定事实 |
| `TIMELINE.md` | 必须 | 连续叙事时间线 |
| `diary/` | 推荐 | 每日日记独立目录（启用 `cfire-artist-daily` 技能时维护） |
| `AGENTS.md` | 继承 | 运行准则（从通用模板继承，可追加） |
| `TOOLS.md` | 按需 | 工具协同（无发布能力时可省略） |
| `HEARTBEAT.md` | 按需 | 心跳机制（无主动发布需求时可省略） |
| `REVIEW.md` | 继承 | 审查机制（从通用模板继承） |

### diary/ 独立日记目录（推荐）

日记采用"每日一文件"的独立存储，作为内容生成的权威引用源。

- 目录：`<profile>/diary/`
- 文件命名：`YYYY-MM-DD.md`（严格）
- 文件名必须匹配 `^[0-9]{4}-[0-9]{2}-[0-9]{2}\.md$`
- 同一日期只允许一个文件；重复保存视为更新，覆盖原文件。
- 正文长度 ≤ 200 字（含标点与空格）。
- 内容结构如下（必需章节按顺序存在，`#` 与 `##` 标题精确匹配）：

```markdown
# 日记 / YYYY-MM-DD

## 日期
YYYY-MM-DD

## 情绪
<情绪标签>

## 正文
<日记正文，第一人称，不超过 200 字>

## 元信息
- 字数：<N>
- 保存时间：<ISO 时间，可空>
```

### 情绪标签白名单

仅允许以下值之一：开心、期待、紧张、疲惫、温柔、平静、焦虑、安心、兴奋、迷茫。

### diary Schema 验证规则

1. **命名**：文件名必须是 `YYYY-MM-DD.md`，日期必须合法（2000-01-01 ~ 2100-12-31）。
2. **必需章节**：文件必须包含 `## 日期`、`## 情绪`、`## 正文`，顺序不限。
3. **日期一致性**：`## 日期` 内容必须与文件名中的日期一致。
4. **情绪标签**：必须属于上述白名单。
5. **正文长度**：`## 正文` 去首尾空白后长度 1 ~ 200 字。
6. **无空文件**：`## 正文` 不能为空字符串。

### DIARY.md（可选汇总）

若仍需要一个人类快速浏览入口，可维护 `reference/DIARY.md`。它只是 diary/ 的可选镜像，不参与校验与存储主流程。

## Schema 验证规则

创建或更新艺人档案时，逐项检查：

1. 每个必须文件存在且包含所有必需章节。
2. 列表类型字段至少有最小条目数。
3. 表格类型字段至少有表头和一行示例。
4. 无编造数据——用户未提供的字段标记为"待补充"。
5. `IDENTITY.md` 的"不可随意变更项"非空。
6. `BOUNDARIES.md` 的"绝对禁止"至少 6 条。
7. `VOICE_STYLE.md` 的"禁用表达"至少 3 条。
8. `CONTENT_STRATEGY.md` 的"内容支柱"至少 3 个。
