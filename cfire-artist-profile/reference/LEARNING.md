# LEARNING

## 用途

定义艺人智能体的学习循环机制。让智能体从每次执行结果中积累经验、优化表达、调整策略，使同一艺人的内容质量和人设一致性随使用而持续提升。

## 设计原则

- **外部记忆进化**：不修改模型参数，而是通过更新配置文件和记忆数据来积累经验。
- **紧凑聚焦**：记忆只保存会反复使用的信息，不堆积一次性上下文。
- **人机协作**：涉及人设变更、表达风格调整、边界修订的学习结果必须人工确认后才写入。
- **可追溯**：每次学习写入都记录来源和原因，支持后续审计和回滚。

## 学习触发点

| 触发场景 | 学习类型 | 目标文件 |
| --- | --- | --- |
| 内容发布后获得正面反馈 | 正向强化 | `MEMORY.md`（内容经验） |
| 内容发布后获得负面反馈 | 负向修正 | `VOICE_STYLE.md` 或 `REVIEW.md` |
| 用户修改了智能体生成的草稿 | 表达校准 | `VOICE_STYLE.md`（常用/禁用表达） |
| 用户拒绝了某条内容建议 | 策略调整 | `CONTENT_STRATEGY.md`（生成规则） |
| 粉丝互动中发现新的偏好模式 | 粉丝理解 | `FAN_RELATIONSHIP.md` |
| 活动复盘发现流程优化点 | 流程优化 | `REVIEW.md`（检查清单） |
| 时间线事件被反复引用 | 记忆沉淀 | `MEMORY.md`（从 TIMELINE 提炼） |
| 发现人设漂移或表达不一致 | 一致性修复 | `SOUL.md` 或 `IDENTITY.md` |
| 用户补充了新的背景信息 | 知识扩充 | `IDENTITY.md` 或 `MEMORY.md` |
| 连续多次生成同类内容 | 多样性优化 | `CONTENT_STRATEGY.md`（内容支柱） |

## 学习循环流程

### 1. 执行后评估

每次完成任务（内容生成、粉丝回复、活动复盘等）后，自动检查：

```
- 用户是否修改了草稿？修改了什么？
- 用户是否拒绝了建议？拒绝原因是什么？
- 内容发布后是否有可观测的反馈信号？
- 是否与已有记忆或时间线产生冲突？
- 是否有重复模式需要打破？
```

### 2. 学习候选生成

根据评估结果，生成学习候选：

```markdown
## 学习候选
- 来源：<触发场景描述>
- 类型：<正向强化 / 负向修正 / 策略调整 / 记忆沉淀 / 一致性修复>
- 建议变更：<具体变更内容>
- 目标文件：<要更新的文件>
- 影响范围：<影响的后续行为>
- 是否需确认：<是 / 否>
```

### 3. 变更分级与确认

| 变更级别 | 示例 | 确认要求 |
| --- | --- | --- |
| L0 - 自动写入 | 时间线事件沉淀到长期记忆 | 无需确认 |
| L1 - 通知写入 | 粉丝偏好记录、内容经验积累 | 写入后通知用户 |
| L2 - 确认后写入 | 语气微调、内容策略调整、禁用表达新增 | 必须用户确认 |
| L3 - 提案审核 | 人设修改、价值观调整、边界变更 | 展示完整变更方案，用户审核 |

### 4. 写入与记录

确认后执行写入，同时在学习日志中记录：

```markdown
| 时间 | 触发场景 | 变更类型 | 目标文件 | 变更摘要 | 确认级别 |
| --- | --- | --- | --- | --- | --- |
```

## 周期性学习

### 每周回顾

每周心跳检查时，额外执行：

1. **记忆整理**：从 `TIMELINE` 中提炼反复引用的事件到 `MEMORY`。
2. **重复度检查**：最近 7 天内容是否过度集中在某个支柱或角度。
3. **人设一致性**：最近输出是否偏离 `SOUL.md` 定义的性格和价值观。
4. **表达新鲜度**：`VOICE_STYLE.md` 的常用表达是否需要补充新场景。

### 每月回顾

每月首次心跳时，建议：

1. **目标对齐**：`GOALS.md` 的阶段目标是否需要更新。
2. **策略优化**：`CONTENT_STRATEGY.md` 的内容支柱和节奏是否需要调整。
3. **边界审视**：`BOUNDARIES.md` 是否有需要新增或修订的边界。
4. **粉丝关系**：`FAN_RELATIONSHIP.md` 的互动模式是否需要演化。

## 学习日志格式

学习日志保存在本文件下方的「学习记录」表格中，同时以摘要形式附加到 `MEMORY.md` 底部。

```markdown
## 学习记录

| 时间 | 来源 | 类型 | 变更摘要 | 目标文件 | 级别 |
| --- | --- | --- | --- | --- | --- |
```

## 防止过拟合

- 单次反馈不轻易改变整体策略，至少 3 次同类信号才触发策略级调整。
- 正向强化只记录经验，不自动扩大同类内容比例。
- 负向修正优先检查是个例还是模式，个例记录到 `REVIEW.md`，模式才修改策略。
- 人设核心（`IDENTITY.md` 的不可变更项、`SOUL.md` 的性格底色）不因单次反馈而修改。
- 学习结果之间如有冲突，以时间近的为准，但安全边界永远不被学习覆盖。

## 与 memory_store 的协同

学习循环以 Markdown 文件为权威存储：

- 学习候选 → 先以"建议写入"输出，记录在 `MEMORY.md` 或 `REVIEW.md`
- 确认写入 → 直接编辑对应 Markdown 文件（L2+ 需用户确认）
- 学习日志 → 记录在上文「学习记录」表格中
- 周期回顾 → 读取 `MEMORY.md` 与 `cfire-artist-daily` 时间线，必要时用 `search.get_recent_context()` 拉取近期上下文
