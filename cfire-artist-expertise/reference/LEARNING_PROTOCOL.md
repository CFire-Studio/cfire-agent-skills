# LEARNING_PROTOCOL

## 用途

定义智能体自主学习专业技能的完整流程，包括技能差距识别、知识获取、内容整理、确认保存和定期更新。

## 设计原则

- **按需学习**：只在任务需求明确超出已有技能范围时触发学习
- **来源可靠**：优先选择权威来源（学术论文、专业书籍、行业标准）
- **结构规范**：学习成果必须遵循 `EXPERTISE_SCHEMA.md` 规范
- **人机协作**：智能体可自主起草技能文档，但正式保存前必须用户确认
- **持续更新**：技能文档应定期审查和更新，保持时效性

## 学习触发点

| 触发场景 | 触发条件 | 学习类型 |
| --- | --- | --- |
| 任务需求超出已有技能 | 当前任务涉及的领域在 `EXPERTISE_INDEX.md` 中不存在相关技能 | 新知识获取 |
| 技能内容不足 | 已有技能文档无法满足当前任务的深度需求 | 技能深化 |
| 用户明确要求学习 | 用户直接要求智能体学习某项技能 | 定向学习 |
| 技能时效性过期 | 技能文档的 `update_date` 超过 6 个月未更新 | 技能更新 |
| 技能应用失败 | 应用已有技能完成任务时出现明显错误或不足 | 技能修正 |

## 学习流程

### 阶段一：技能差距识别

1. **任务分析**：分析当前任务的需求，提取关键领域和技能要求
2. **技能检索**：通过 `EXPERTISE_INDEX.md` 检索已有技能
3. **差距评估**：对比任务需求与已有技能，确定差距程度

#### 差距评估标准

| 差距程度 | 说明 | 处理方式 |
| --- | --- | --- |
| 无差距 | 已有技能完全满足需求 | 直接应用已有技能 |
| 轻微差距 | 已有技能基本满足，需少量补充 | 在已有技能基础上更新 |
| 中度差距 | 已有技能部分满足，需补充核心知识 | 创建新版本或补充章节 |
| 重大差距 | 完全没有相关技能或技能严重过时 | 全新学习并创建技能文档 |

### 阶段二：知识获取

根据差距评估结果，通过以下方式获取知识：

#### 信息收集方法

| 方法 | 适用场景 | 可靠性 |
| --- | --- | --- |
| 互联网搜索 | 获取最新行业动态和实践经验 | 中等，需验证 |
| 学术论文检索 | 获取专业理论和研究成果 | 高 |
| 专业书籍阅读 | 获取系统知识体系 | 高 |
| 用户输入 | 获取个性化知识和经验 | 中，需确认 |
| 行业报告分析 | 获取行业趋势和数据 | 高 |

#### 信息验证原则

1. 优先选择权威来源（学术期刊、专业机构、知名专家）
2. 交叉验证多个来源的一致性
3. 标注信息来源，便于后续追溯
4. 对有争议的观点明确标注

### 阶段三：内容整理

将收集到的知识整理成符合 `EXPERTISE_SCHEMA.md` 规范的技能文档：

1. **确定技能名称和领域**：根据知识内容确定技能名称和所属领域
2. **编写元数据头部**：填写 `name`、`domain`、`proficiency`、`acquisition_date`、`source` 等字段
3. **组织知识结构**：按照推荐模板组织技能主体内容
4. **添加标签**：根据内容添加 3-5 个检索标签
5. **自我校验**：检查是否符合 Schema 校验项

### 阶段四：确认与保存

技能文档创建完成后，进入确认流程：

#### 确认流程

```
智能体起草技能文档
    ↓
生成学习候选报告
    ↓
提交用户确认（L2 级别）
    ↓
┌────────────┐
│ 用户确认   │──是──→ 保存技能文档到 assets/expertise/
│           │              更新 EXPERTISE_INDEX.md
└────────────┘              记录学习日志
    ↓否
┌────────────┐
│ 用户拒绝   │──→ 返回修改，重新起草
└────────────┘
```

#### 学习候选报告格式

```markdown
## 学习候选
- 来源：<触发场景描述>
- 类型：<新知识获取 / 技能深化 / 定向学习 / 技能更新 / 技能修正>
- 技能名称：<技能名称>
- 技能领域：<技能领域>
- 建议熟练度：<入门 / 初级 / 中级 / 高级 / 专家>
- 知识来源：<主要来源列表>
- 影响范围：<该技能将影响哪些后续任务和行为>
- 是否需确认：是（L2 级别）
- 技能文档预览：
  <技能文档摘要或关键内容>
```

#### 变更级别

| 变更级别 | 示例 | 确认要求 |
| --- | --- | --- |
| L0 - 自动记录 | 更新技能文档的参考资料 | 无需确认 |
| L1 - 通知写入 | 补充技能文档的次要章节 | 写入后通知用户 |
| L2 - 确认后写入 | 创建新技能文档、更新核心知识 | 必须用户确认 |
| L3 - 提案审核 | 大规模技能体系重构 | 展示完整方案，用户审核 |

### 阶段五：技能应用与反馈

技能文档保存后，进入应用和反馈阶段：

1. **技能应用**：在后续相关任务中应用新学习的技能
2. **效果评估**：观察技能应用的效果，收集反馈
3. **技能优化**：根据反馈结果优化技能文档内容
4. **周期更新**：定期审查和更新技能文档

## 周期性学习

### 每月回顾

每月首次心跳检查时，执行：

1. **技能时效性检查**：检查所有技能文档的 `update_date`，超过 6 个月未更新的标记为"需更新"
2. **技能应用频率分析**：分析哪些技能被频繁使用，哪些几乎未被使用
3. **技能差距重新评估**：根据近期任务类型，评估是否需要学习新技能
4. **技能优先级排序**：根据应用频率和重要性，确定技能更新的优先级

### 季度回顾

每季度执行：

1. **技能体系完整性评估**：检查当前技能体系是否覆盖艺人主要工作领域
2. **技能关联性分析**：分析技能之间的关联，发现知识缺口
3. **技能版本管理**：审查技能版本，合并或清理过时版本

## 学习日志

学习日志记录每次学习活动，格式如下：

```markdown
| 时间 | 触发场景 | 学习类型 | 技能名称 | 变更级别 | 结果 |
| --- | --- | --- | --- | --- | --- |
| 2026-07-02 | 任务需求超出已有技能 | 新知识获取 | 音乐疗愈技能 | L2 | 已确认并保存 |
```

学习日志保存在 `memory_store` 的 `learning_log` 表中，同时以摘要形式附加到技能文档的底部。

## 防止学习偏差

- 单次学习不轻易改变技能核心内容，至少 3 次同类反馈才触发核心更新
- 学习内容必须与艺人设定和目标一致，避免学习无关或冲突的技能
- 技能文档应保持专业性和客观性，避免主观臆断
- 学习过程应记录完整，便于后续审计和追溯

## 与 cfire-artist-profile 的协同

学习流程与 `cfire-artist-profile` 的 `LEARNING.md` 协同：

- 学习触发点与 `LEARNING.md` 的学习触发点对齐
- 变更级别与 `LEARNING.md` 的变更分级一致
- 学习日志统一存储在 `memory_store`
- 技能学习结果可沉淀到 `MEMORY.md` 作为长期记忆
